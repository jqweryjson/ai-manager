# Telegram Listener — Инструкция по запуску

## Описание

Telegram Listener — это фоновый воркер, который прослушивает новые сообщения в подключенных Telegram чатах и выводит их в консоль (в будущем — генерирует и отправляет автоответы через LLM).

## Архитектура

- **Один Node.js процесс** с множеством `TelegramClient` объектов
- Каждый аккаунт = один клиент = ~5-10 МБ RAM + 1 TCP соединение
- Автоматическое переподключение при сетевых ошибках
- Graceful shutdown при SIGTERM/SIGINT
- Docker + restart policy для автоматического перезапуска при падении

## Требования

- PostgreSQL с таблицами `telegram_accounts` и `telegram_subscriptions`
- Переменные окружения:
  - `DATABASE_URL` — строка подключения к PostgreSQL
  - `JWT_SECRET` — секрет для JWT
  - `BACKEND_SECRET` — ключ для AES-256-GCM шифрования

```bash
npm run build
```

## Запуск в Docker (production)

### 1. Сборка образа

```bash
docker-compose build telegram-listener
```

### 2. Запуск сервиса

```bash
docker-compose up -d telegram-listener
```

### 3. Просмотр логов

```bash
docker-compose logs -f telegram-listener
```

### 4. Остановка сервиса

```bash
docker-compose stop telegram-listener
```

### 5. Перезапуск сервиса

```bash
docker-compose restart telegram-listener
```

### 6. Проверка статуса

```bash
docker-compose ps telegram-listener
```

## Как это работает

### 1. Инициализация при старте

- Воркер загружает из БД все аккаунты со статусом `connected`
- Для каждого аккаунта проверяется наличие хотя бы одной `enabled` подписки
- Создается `TelegramClient` с расшифрованной сессией
- Клиент подключается к Telegram и подписывается на события `NewMessage`

### 2. Обработка новых сообщений

- При получении нового сообщения проверяется, есть ли `enabled` подписка для этого чата
- Если есть — сообщение выводится в консоль:
  ```
  ============================================================
  📨 НОВОЕ СООБЩЕНИЕ
  ============================================================
  🔹 Чат: Семья (chat)
  🔹 Peer ID: 1234567890
  🔹 Sender ID: 9876543210
  🔹 Текст: Привет, как дела?
  🔹 Workspace: ws_1234567890
  🔹 Role: role_1234567890
  🔹 Дата: 14.11.2025, 12:30:45
  ============================================================
  ```
- Если нет — сообщение игнорируется

### 3. Переподключение при сетевых ошибках

- GramJS автоматически переподключается при потере соединения
- Бесконечные попытки переподключения (каждую 1 сек)
- Telegram доставляет пропущенные сообщения при восстановлении соединения

### 4. Обработка ошибок

- Любая ошибка в обработчике сообщения логируется, но не останавливает воркер
- При сбое запуска слушателя для аккаунта — повторная попытка через 30 сек
- `uncaughtException` и `unhandledRejection` логируются, но не останавливают процесс

### 5. Graceful shutdown

- При получении SIGTERM (Docker stop) или SIGINT (Ctrl+C):
  1. Отключаются все клиенты
  2. Закрывается пул БД
  3. Процесс завершается с кодом 0

## Масштабирование

### Малые объемы (1-100 аккаунтов)

- **RAM**: ~500 МБ - 1 ГБ
- **CPU**: минимальная
- **Сервер**: $5-10/месяц

### Средние объемы (100-1000 аккаунтов)

- **RAM**: ~2-4 ГБ
- **CPU**: 1-2 cores
- **Сервер**: $20-40/месяц

### Большие объемы (1000+ аккаунтов)

- Горизонтальное масштабирование (несколько серверов)
- Партиционирование: 500 аккаунтов на сервер
- Балансировка нагрузки

## Troubleshooting

### Проблема: Воркер не запускается

**Проверьте:**

1. Переменные окружения установлены корректно
2. PostgreSQL доступен
3. Таблицы `telegram_accounts` и `telegram_subscriptions` созданы
4. Логи: `docker-compose logs telegram-listener`

### Проблема: Не прослушиваются сообщения

**Проверьте:**

1. Аккаунт в статусе `connected` (не `pending_code` или `pending_2fa`)
2. Есть хотя бы одна подписка с `enabled = true`
3. Правильные `workspace_id` и `role_id` в подписке
4. Логи: есть ли сообщение "✅ Слушатель запущен для..."

### Проблема: Воркер часто падает и перезапускается

**Проверьте:**

1. Логи ошибок: `docker-compose logs telegram-listener`
2. Ресурсы сервера (RAM, CPU)
3. Сетевое соединение (может быть firewall блокирует Telegram)

### Проблема: TIMEOUT ошибки от GramJS

**Решение:** Уже исправлено через `useWSS: false` и `timeout: 10` в конфигурации клиента.

## Архитектурные решения

### Почему один процесс с множеством клиентов?

- **Эффективность**: меньше overhead, чем N процессов
- **Стандартный подход**: WhatsApp Business API, Slack bots используют аналогично
- **Масштабируемость**: покрывает 100-1000 пользователей без проблем

### Почему нельзя один клиент для всех аккаунтов?

MTProto требует отдельную сессию для каждого пользователя. Каждая сессия привязана к уникальному `auth_key`.

### Почему Docker + restart policy?

- **Автоматический перезапуск** при крахе процесса
- **Изоляция** от основного backend сервера
- **Простота развертывания** и масштабирования
- **Production-ready** подход
