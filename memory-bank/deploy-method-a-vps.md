# Продакшен‑деплой по методу A (один VPS + Docker)

Этот стрим описывает, как развернуть текущее приложение (Fastify backend, React/Vite frontend, PostgreSQL, Redis, Milvus, embeddings‑воркер) на одном VPS по **методу A**:

- один VPS‑сервер;
- все сервисы — в Docker / Docker Compose;
- без managed‑сервисов, включая PostgreSQL (Postgres крутится в контейнере).

Подходит для: MVP, небольшого онлайна, старта в проде с минимальной абоненткой.

---

## 1. Требования к VPS и инфраструктуре

**Минимальная конфигурация VPS:**

- **CPU**: 4 vCPU;
- **RAM**: 8 GB;
- **Disk**: 120–160 GB SSD (под PostgreSQL, Milvus и логи);
- **OS**: Ubuntu 22.04 LTS (или другая актуальная LTS);
- **Сеть**:
  - белый IPv4;
  - открытые порты 80/443 (HTTP/HTTPS);
  - SSH‑доступ.

**Сервисы, которые будут крутиться в Docker:**

- `backend` — Fastify API (порт 4001);
- `frontend` — React/Vite (готовая статика, отдаётся через nginx);
- `postgres` — PostgreSQL (БД приложения);
- `redis` — кэш и сессии;
- `milvus` — векторное хранилище;
- `embeddings-worker` — Python‑воркер с bge-m3;
- `nginx` (или `traefik`) — reverse‑proxy и статика фронта.

---

## 2. Подготовка VPS

1. **Создать VPS** у провайдера (Timeweb Cloud, Selectel, RuVDS, FirstVDS, REG.RU и т.п.) с конфигурацией как выше.
2. **Зайти по SSH** под пользователем `root` или sudo‑пользователем.
3. **Обновить систему:**

```bash
sudo apt update && sudo apt upgrade -y
```

4. **Установить Docker и Docker Compose:**

Можно через официальный скрипт Docker или через пакетный менеджер. Пример (через скрипт):

```bash
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER
```

Выход/вход в SSH после добавления в группу `docker`.

При необходимости установить `docker compose` V2 (если не приехал вместе с Docker).

5. **Создать каталог проекта:**

```bash
mkdir -p /opt/ai-manager
cd /opt/ai-manager
```

6. **Клонировать репозиторий:**

```bash
git clone <URL_репозитория> .
```

---

## 3. Настройка переменных окружения (prod)

Создаём отдельные файлы окружения для продакшена (примерная схема, имена можно адаптировать под текущий репозиторий):

- `.env.prod.backend` — для Fastify;
- `.env.prod.frontend` — для Vite/фронта;
- `.env.prod.db` — для PostgreSQL/Milvus/Redis при необходимости.

Ключевые моменты:

**Backend (Fastify):**

- `PORT=4001`;
- `NODE_ENV=production`;
- JWT/секреты;
- креды к PostgreSQL (host, port, db, user, password);
- параметры Milvus, Redis, embeddings‑воркера;
- Groq API key.

**Frontend (Vite):**

- `VITE_API_BASE_URL=https://<твой_домен>/api`;

**PostgreSQL (контейнер):**

- `POSTGRES_DB`;
- `POSTGRES_USER`;
- `POSTGRES_PASSWORD`.

Эти переменные будут проброшены в `docker-compose.prod.yml`.

---

## 4. Docker Compose для продакшена

Создаём файл `docker-compose.prod.yml` (имя условное, можно использовать уже существующий, если он есть, но ниже логика именно метода A):

- сервис `backend`:
  - собирается из `Dockerfile` backend’а;
  - зависит от `postgres`, `redis`, `milvus`, `embeddings-worker`;
  - слушает порт 4001 (пробрасывается только внутрь сети Docker, наружу ходит nginx).

- сервис `frontend-build` или `frontend`:
  - билд фронта (`npm ci && npm run build`);
  - статика монтируется в nginx.

- сервис `nginx`:
  - слушает 80/443;
  - отдаёт статику фронта;
  - проксирует `/api` → `backend:4001`.

- сервис `postgres`:
  - официальный образ Postgres;
  - volume для данных (`postgres_data`);
  - env для БД.

- сервис `redis`:
  - официальный образ Redis;
  - volume под данные (по желанию).

- сервис `milvus` и вспомогательные контейнеры:
  - как в dev/compose‑конфиге (etcd, minio и т.п., если используются);
  - отдельный volume под данные (`milvus_data`).

- сервис `embeddings-worker`:
  - Python контейнер с моделью bge-m3;
  - env для подключения к Milvus и т.д.

Важно: все сервисы объединяем в общую docker‑сеть (например, `ai-manager-net`).

---

## 5. Сборка и запуск

1. **Сборка образов:**

```bash
docker compose -f docker-compose.prod.yml build
```

2. **Запуск в фоне:**

```bash
docker compose -f docker-compose.prod.yml up -d
```

3. **Проверка:**

- посмотреть статусы:

```bash
docker compose -f docker-compose.prod.yml ps
```

- логи backend’а:

```bash
docker compose -f docker-compose.prod.yml logs backend -f
```

---

## 6. Nginx и HTTPS

Если используем nginx как часть `docker-compose`:

- в конфиге nginx прописываем:
  - `location /` — статика фронта (`/usr/share/nginx/html` или volume с билдом Vite);
  - `location /api/` — прокси на `http://backend:4001`.

Для HTTPS:

- можно использовать `certbot` внутри отдельного nginx‑контейнера или генерировать сертификаты снаружи и монтировать их в контейнер.
- упрощённый вариант на старте — HTTP без TLS, а TLS добавить позже (но лучше сразу сделать HTTPS).

## 7. Бэкапы и обновления (метод A)

### Бэкапы

**PostgreSQL (в контейнере):**

- регулярный `pg_dump` (скрипт/cron на VPS), пример:

```bash
docker exec <postgres_container_name> \
  pg_dump -U <user> -d <db> > /opt/backups/db_$(date +%F).sql
```

- хранить дампы:
  - локально (каталог `/opt/backups`);
  - и/или отправлять в облачное хранилище (S3/облако провайдера).

**Milvus:**

- бэкапить docker‑тома (`milvus_data`), например через `tar` или snapshot’ы диска на уровне провайдера.

### Обновления приложения

Типичный цикл:

```bash
cd /opt/ai-manager
git pull
docker compose -f docker-compose.prod.yml build
docker compose -f docker-compose.prod.yml up -d
```

При необходимости предварительно прогонять миграции БД, если они есть.

## 8. Что такое managed PostgreSQL и чем он отличается от Postgres в контейнере

### Managed PostgreSQL — что это

**Managed PostgreSQL** — это когда облачный провайдер (Timeweb Cloud, Selectel, Yandex Cloud и т.д.) предоставляет тебе PostgreSQL как готовый сервис:

- провайдер:
  - устанавливает и настраивает PostgreSQL;
  - ставит обновления и патчи безопасности;
  - делает автоматические бэкапы и снапшоты;
  - часто даёт репликацию, фейловер, мониторинг «из коробки».
- ты получаешь:
  - хост, порт, имя БД, логин, пароль;
  - не занимаешься администрированием самого postgres‑сервера.

Фактически это **“PostgreSQL по подписке”**: ты платишь чуть больше, но экономишь время и нервы на админке.

### PostgreSQL в контейнере (как в методе A)

В методе A PostgreSQL крутится как обычный сервис в `docker-compose`:

- используем официальный Docker‑образ `postgres`;
- данные лежат в Docker volume (`postgres_data`);
- все обязанности по администрированию на тебе:
  - настройка параметров (память, журналы, лимиты, max_connections и т.п.);
  - обновление версии Postgres (перенос данных при смене образа);
  - организация бэкапов (`pg_dump`, snapshot’ы, проверка восстановления);
  - мониторинг нагрузки и ошибок.

Плюсы:

- дешевле по деньгам (платишь только за VPS);
- всё живёт в одном месте (один `docker-compose`, один сервер).

Минусы:

- больше ручной работы и зон ответственности;
- риск потери данных выше, если не организовать бэкапы и не следить за диском/ресурсами.

### Ключевые отличия managed PostgreSQL vs Postgres в контейнере

**1. Управление и поддержка**

- Managed:
  - управляет провайдер;
  - обновления, патчи, бэкапы, реплики — их зона ответственности.
- Контейнер:
  - управляешь ты;
  - всё, что касается надёжности и апдейтов, нужно продумать и реализовать самостоятельно.

**2. Надёжность и SLA**

- Managed:
  - обычно есть официальный SLA, отказоустойчивость, мультизонность;
  - бэкапы и восстановление — стандартная функция сервиса.
- Контейнер:
  - SLA зависит только от настройки VPS и твоих скриптов;
  - если забыл настроить бэкапы или сломал конфиг — отвечать тоже тебе.

**3. Стоимость**

- Managed:
  - дороже при тех же ресурсах (платишь за сервис и автоматизацию);
  - выгодно, когда ценно именно время/надёжность, а не каждый рубль железа.
- Контейнер:
  - дешевле, платишь только за сам VPS;
  - хорошо для MVP, тестов, небольшого продакшена.

**4. Когда выбирать что**

- **Метод A (Postgres в контейнере)**:
  - MVP / ранний прод;
  - ограниченный бюджет;
  - небольшой онлайн и объём данных;
  - есть готовность руками настроить бэкапы и следить за диском.

- **Managed PostgreSQL**:
  - растущий продукт, появляется выручка и требования к SLA;
  - критично не терять данные и быстро восстанавливаться;
  - хочется меньше заниматься администрированием и сосредоточиться на фичах.

## 9. Миграция с контейнерного Postgres на managed PostgreSQL (в будущее)

Когда вырастет нагрузка и появится необходимость в большей надёжности:

1. Поднять экземпляр managed PostgreSQL у провайдера.
2. Сделать `pg_dump` текущей базы (контейнерный Postgres) и залить дамп в managed:

```bash
pg_dump -h <old_host> -U <user> -d <db> > dump.sql
psql -h <managed_host> -U <user> -d <db> < dump.sql
```

3. Обновить переменные окружения backend’а (DSN БД на managed хост).
4. Перезапустить backend (через `docker compose up -d`).
5. После проверки — выключить контейнерный Postgres и почистить volume, когда убедишься, что миграция удалась.

Метод A даёт быстрый старт за счёт Postgres в контейнере, а при росте можно относительно безболезненно перейти на managed PostgreSQL, не меняя остальную архитектуру.
