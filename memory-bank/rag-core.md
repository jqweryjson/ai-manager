# RAG: глоссарий и верхнеуровневые шаги

Дата: 09.01.2025

Источник подхода: RAG архитектура и практика из статьи и обсуждений — [Построение базы знаний компании и поиска документов на LLM и RAG](https://habr.com/ru/companies/raft/articles/863888/) и [комментарии](https://habr.com/ru/companies/raft/articles/863888/comments/).

## Глоссарий (краткие определения)

- Retrieval: поиск релевантных фрагментов в индексе (Milvus), обычно top_k с метрикой (cosine/IP) и фильтрами.
- Chunking: разбиение документов на смысловые куски для индексирования и последующего поиска.
- Embeddings: преобразование текста в векторы фиксированной размерности (например, bge-m3, 1024 dim).
- ANN Index: структура для быстрого поиска ближайших соседей (HNSW/IVF) по выбранной метрике.
- Hybrid Search: объединение dense (вектора) и sparse (BM25) результатов, последующий фьюжн.
- Rerank: переклассификация кандидатов с cross-encoder для повышения точности топ-N.
- Query Rewrite: нормализация/переформулирование запроса для улучшения ретрива.
- Diversification: обеспечение разнообразия источников (группировка по doc_id, ограничение чанков на документ).
- Dedup: удаление дублей/почти-дублей (например, по hash/SimHash) на этапе ingestion или ретрива.
- Coverage: оценка доли ответа, реально опирающейся на предоставленный контекст [0..1].
- Citations: ссылки на источники (doc_id, chunk_id, snippet/source) для проверки пользователем.
- Prompt Builder: формирование строгих инструкций, формата ответа и токен-бюджета для LLM.
- JSON Schema: требование вернуть структурированный JSON (answer, citations[], coverage), валидация перед выдачей.
- Self-check: самопроверка/верификация ответа моделью (переформулировка, consistency-check, оценка уверенности).
- Tool-use: выполнение вычислений/запросов вне LLM (SQL/Elasticsearch/Pandas) и подмешивание результатов.
- Guardrails: политики компании и ограничители (что можно/нельзя отвечать, стиль, тон, запреты на «вымысел»).
- Cache: кэширование ответов (Redis) по ключу «нормализованный запрос + версия корпуса» с TTL.
- Corpus Versioning: версионность корпусов и атомарная публикация/rollback индексов.
- Observability: тайминги, метрики качества (coverage, % цитат), трассировка (reqId) и логи на этапах пайплайна.
- Data Isolation: строгие фильтры по user_id на всех слоях (ingest, retrieve, access).
- Streaming: потоковая выдача ответа (токенов) на клиент.
- Cost Controls: выбор дешёвых/дорогих моделей по триггерам, лимиты токенов/времени/ретраев.

## Верхнеуровневые шаги RAG → LLM (ядро)

1. Normalize: нормализовать/переписать запрос; извлечь сигналы (язык, намерение).
2. Retrieve: dense-поиск в Milvus (top_k, cosine/IP) с фильтром по user_id.
3. (Опц.) Hybrid fuse: добавить sparse (BM25) и объединить результаты.
4. (Опц.) Rerank: cross-encoder до top_8 по релевантности.
5. Select context: дедуп, диверсификация по doc_id, лимит чанков на документ, общий токен-бюджет.
6. Build prompt: строгие инструкции «только по контенту», формат JSON, ограничения и guardrails.
7. Generate (LLM): вызов LLM (Groq), temperature/top_p/max_tokens, при необходимости стриминг.
8. Validate: проверка JSON‑схемы, оценка coverage ≥ threshold; при недостатке — честный отказ/уточнение.
9. Return: ответ + citations + coverage; логирование размеров и таймингов.
10. Cache: записать результат в Redis с TTL и версией корпуса.
11. Metrics: собрать метрики (p50/p95, % цитат), трассировка по reqId.

## MVP → Рост

- MVP: dense retrieval, prompt builder, Groq генерация, JSON+citations, cache, базовые метрики.
- Рост: hybrid BM25, rerank, self-check, инструменты для вычислений, версионность корпусов, расширенная наблюдаемость.
