## Редактируемая база знаний (Editable Knowledge Stream)

### Цели

- **Точечные правки** знаний без перелития всего корпуса: обновляем только затронутые чанки.
- **Быстрая обратная связь из UI**: «карандаш» на ответе → правка цитат → сохранить → мгновенно влияет на последующие ответы.
- **Аудит и откаты**: храним версионность изменённых фрагментов, можно откатиться.

### Основные сценарии

- Правка устаревших фактов, прозвучавших в ответе, по конкретным цитатам (`chunk_id`).
- Добавление/удаление фактов в пределах одного чанка.
- Локальное перечанкивание вокруг изменённой области (опционально).

### Требования и ограничения

- Изменения должны затрагивать только выбранные чанки (или локальную область при перечанкивании).
- Источник-оригинал (pdf/doc/xlsx/json) остаётся архивным; «истина» хранится в нормализованном тексте базы знаний.
- Необходимо инвалидировать кэши (Redis) по затронутым фрагментам.
- Обеспечить согласованность между Postgres и Milvus при частичных неудачах (идемпотентность, транзакции/сага).

### Модель данных (Postgres)

- `documents(id, user_id, type, source_meta, version, created_at)`
- `document_chunks(id, doc_id, user_id, text, anchor_json, start_offset, end_offset, supersedes_chunk_id NULLABLE, is_active BOOLEAN, embedding_hash, updated_at)`
  - `anchor_json`: якорь положения в источнике: для PDF — `{page, heading, paragraph}`; для Excel — `{sheet, row, col}`; для JSON — `{json_path}`.
  - `supersedes_chunk_id`: связь с предыдущей версией фрагмента.
  - `is_active`: только активные фрагменты участвуют в поиске.

### Индексация (Milvus)

- `vector_id = chunk_id` (строгая привязка).
- Поля метаданных: `{chunk_id, doc_id, user_id, is_active}`.
- Операции: точечный delete по `chunk_id` и insert для нового вектора.

### Процесс правки (E2E)

1. Ответ чата содержит `citations: chunk_id[]`.
2. Пользователь жмёт «карандаш» → фронт запрашивает `GET /api/chunks?ids=...` и показывает редактор.
3. При сохранении:
   - Для каждого изменённого чанка:
     - Создаём новую строку в `document_chunks` (копия + правки текста), старый `is_active=false`, новый `is_active=true`, проставляем `supersedes_chunk_id`.
     - Воркер считает эмбеддинг для нового текста.
     - В Milvus: delete старого `chunk_id`, insert нового с новым `chunk_id`.
   - Инвалидируем Redis‑кэши, где упоминались эти `chunk_id`.

### Перечанкивание (локальное)

- Триггер по флажку «перечанкировать соседние»: берём изменённый диапазон и N соседних фрагментов (N=1..2), объединяем и перестраиваем чанки локально.
- Пересоздаём только затронутые чанки, прочие не трогаем.

### API (черновик)

- `GET /api/chunks?ids=<list>` → `[{chunk_id, doc_id, text, anchor_json, is_active}]`.
- `POST /api/chunks/edit` → body: `{ edits: [{ chunk_id, new_text, local_rechunk?: boolean }] }` → result: `{ replaced: [{old_chunk_id, new_chunk_ids[]}] }`.
- `POST /api/chunks/upsert` → body: `{ doc_id, anchor_json?, text }` → создание нового чанка (для пустых мест).
- `POST /api/chunks/delete` → soft-delete по `chunk_id`.

### Инвалидирование кэша

- Ключи Redis по чат‑сессиям должны хранить ссылку на `chunk_id[]` (или версию корпуса).
- При изменении — удалять записи с участием изменённых `chunk_id` или повышать версию корпуса пользователя.

### Координация и надёжность

- Транзакция в Postgres: запись нового чанка + деактивация старого.
- Сага/ретраи: если Milvus insert/delete не удался, пометка «reindex_required=true» и фоновый ретрай.
- Идемпотентность операций edit по `client_edit_id`.

### Воркер эмбеддингов

- Вход: `{ chunk_id, text }` → выход: `{ chunk_id, vector }`.
- Очередь задач (Redis/AMQP). Повторная попытка при ошибке.

### UI/UX

- Кнопка «карандаш» на ответе → модал редактора.
- Вкладки по цитатам; внутри — поле редактирования нормализованного текста.
- Опции: «Сохранить», «Перечанкировать локально», «Отменить».
- Права: только владелец/workspace‑админ.

### Мультиформатность

- PDF/Word: якоря `{page, heading, paragraph}`.
- Excel: `{sheet, row, col}` и/или диапазоны.
- JSON: `{json_path}`; опционально можно хранить типизированные «факты».
- Ограничение: правки не возвращаются в исходные бинарные файлы; базой истины становится нормализованный слой.

### Риски и меры

- Расхождение с исходником: документируем, что база знаний — рабочая копия.
- Потеря связности при массовых правках: локальный перечанкинг по окрестности.
- Кэш устарел: строгая инвалидация по `chunk_id`/версии.
- Гонки обновлений: optimistic locking по `updated_at` или версиям.

### Этапы внедрения (Roadmap)

1. Схема БД: `document_chunks` с версионированием и якорями. Миграция текущих данных.
2. Эндпоинты: `GET /chunks`, `POST /chunks/edit` (без перечанкивания), инвалидация Redis.
3. Интеграция воркера: пересчёт эмбеддингов для изменённых чанков.
4. Milvus: delete+insert по `chunk_id`. Сага/ретраи.
5. Фронтенд: UI редактора правок из ответа по `citations`.
6. Локальное перечанкивание соседей (N=1) и UI‑флажок.
7. Мультиформат: PDF страницы и JSON json_path якоря.
8. Аудит/откаты и права доступа.

### Критерии готовности (MVP)

- Можно отредактировать цитируемый фрагмент, сохранить и получить новый ответ с учётом правки.
- Посторонние части корпуса не меняются.
- Есть история правок по `supersedes_chunk_id`.
- Инвалидация кэша подтверждена в e2e.

### Метрики

- % ответов, затронутых правками, где ассистент перестал ссылаться на устаревшие факты.
- Среднее время применения правки (UI → индекс обновлён).
- Кол-во ретраев/ошибок в саге Milvus.
